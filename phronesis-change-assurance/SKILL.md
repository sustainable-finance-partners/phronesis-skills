# phronesis-change-assurance

## Purpose
Before an agent applies a high-risk change, call Phronesis to assess whether it is
**defensible enough to apply** — and get a Decision Receipt. Phronesis **consumes** the
change context (diff, SBOM, static-analysis, formal-method output, migration mapping)
from your tools; it **does not apply the change, build the config, or run the
migration** — your tools (Cloudflare, CI/CD, IaC, the migration skill) apply it.

> A change generated by a capable tool is not automatically a **safe** change. Every
> high-risk change needs a diff, an owner, a rollback, and a boundary before it is
> committed. *Generated is not approved.*

## When to use
Before applying: security / infra config changes (Zero Trust, firewall, IAM policy) ·
production deploys · PR merges · dependency acceptance · data exports · vendor migrations.

## Required context (collect before calling)
- `change_context` — `before_state` / `after_state` · `diff_summary` · `blast_radius` ·
  affected users/apps · `owner` · `rollback_plan` (present/partial/absent) · `rollback_tested`
- `control_path_context` — see `phronesis-mcp-control-path`
- `evidence_context` — test / static-analysis / formal-method / SBOM / runtime-validation outputs
- `migration_context` — for vendor migrations: source/target, mapped/unmapped objects, policy equivalence
- `decision_materiality`

## Outputs — and how to act on each
- `approve` — apply; record the Decision Receipt ID.
- `review` — pause; route to the human owner with the diff + rollback.
- `block` — do not apply; surface the reason.
- `stage_only` — stage the change without applying; human confirms before commit.
- `seek_evidence` — e.g. unmatched policy mapping → gather, then re-call.
- `require_human` — high blast radius / no rollback → human approval before apply.

## Anti-patterns (do NOT)
- Do **not** apply a high-risk change without diff, owner, and rollback in `change_context`.
- Do **not** treat a generated change as safe because a capable tool produced it.
- Do **not** skip human review for high-blast-radius or irreversible changes.
- Do **not** apply without control-path context; do **not** trust client-supplied identity.

## Working directory (`.phronesis/`)
Keep the change's trail per the repo convention
([phronesis-working-directory.md](../phronesis-working-directory.md)):
`decision_requirement.md` + `evidence_notes.md` (diff, blast radius, rollback, analysis
outputs), `action_boundary_request.json`, `decision_receipt.json` (verbatim),
`review_log.md` on `review`/`stage_only`/`require_human`, `outcome_followup.md` after the
change lands.

## Review-before-apply (the pattern)
Propose the change + context → call Phronesis → action boundary + Receipt → human review
if `review`/`stage_only`/`require_human` → your tool applies it (**Phronesis never
does**) → record the Receipt ID → outcome scoring closes the loop.
