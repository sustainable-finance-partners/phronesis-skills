# phronesis-change-assurance

## Purpose
Before an agent applies a high-risk change, call Phronesis to assess whether it is
**defensible enough to apply** вЂ” and get a Decision Receipt. Phronesis **consumes** the
change context (diff, SBOM, static-analysis, formal-method output, migration mapping)
from your tools; it **does not apply the change, build the config, or run the
migration** вЂ” your tools (Cloudflare, CI/CD, IaC, the migration skill) apply it.

> A change generated by a capable tool is not automatically a **safe** change. Every
> high-risk change needs a diff, an owner, a rollback, and a boundary before it is
> committed. *Generated is not approved.*

## When to use
Before applying: security / infra config changes (Zero Trust, firewall, IAM policy) В·
production deploys В· PR merges В· dependency acceptance В· data exports В· vendor migrations.

## Required context (collect before calling)
- `change_context` вЂ” `before_state` / `after_state` В· `diff_summary` В· `blast_radius` В·
  affected users/apps В· `owner` В· `rollback_plan` (present/partial/absent) В· `rollback_tested`
- `control_path_context` вЂ” see `phronesis-mcp-control-path`
- `evidence_context` вЂ” test / static-analysis / formal-method / SBOM / runtime-validation outputs
- `migration_context` вЂ” for vendor migrations: source/target, mapped/unmapped objects, policy equivalence
- `decision_materiality`

## Outputs вЂ” and how to act on each
- `approve` вЂ” apply; record the Decision Receipt ID.
- `review` вЂ” pause; route to the human owner with the diff + rollback.
- `block` вЂ” do not apply; surface the reason.
- `stage_only` вЂ” stage the change without applying; human confirms before commit.
- `seek_evidence` вЂ” e.g. unmatched policy mapping в†’ gather, then re-call.
- `require_human` вЂ” high blast radius / no rollback в†’ human approval before apply.

## Anti-patterns (do NOT)
- Do **not** apply a high-risk change without diff, owner, and rollback in `change_context`.
- Do **not** treat a generated change as safe because a capable tool produced it.
- Do **not** skip human review for high-blast-radius or irreversible changes.
- Do **not** apply without control-path context; do **not** trust client-supplied identity.

## Review-before-apply (the pattern)
Propose the change + context в†’ call Phronesis в†’ action boundary + Receipt в†’ human review
if `review`/`stage_only`/`require_human` в†’ your tool applies it (**Phronesis never
does**) в†’ record the Receipt ID в†’ outcome scoring closes the loop.
